# Отчет о дополнительных инструментах разработки

## Проект: CMS онлайн-записи барбершопа «Сайтама»

Основной технологический стек проекта — Python (FastAPI) + Vue 3 + PostgreSQL — обоснован в отдельном документе (`tech_stack_analysis_barbershop.md`). Настоящий отчет описывает **дополнительные библиотеки, плагины и инструменты**, выбранные для решения конкретных производственных задач проекта: миграции схемы БД, аутентификация, хранение паролей, управление состоянием SPA, HTTP-взаимодействие и стилизация интерфейса. Все фрагменты кода взяты из реальной кодовой базы проекта.

## 1. Alembic

### 1.1. Название и Тип
**Alembic** (версия 1.18). Тип: инструмент управления миграциями базы данных для Python. Модуль Python, официальный компаньон ORM SQLAlchemy. Разработчик: Майк Байер (Mike Bayer) и сообщество SQLAlchemy.

### 1.2. Цель и Функциональность
**Главная задача:** версионирование схемы базы данных. Alembic решает проблему эволюции схемы: каждое изменение (новая таблица, колонка, индекс) оформляется как пронумерованный скрипт миграции, который можно применить (`upgrade`) или откатить (`downgrade`) на любом окружении — от локальной машины разработчика до продакшена, — гарантируя идентичность схемы.

**Ключевые возможности (использованные в проекте):**
- **Автогенерация миграций** (`alembic revision --autogenerate`) — сравнение ORM-моделей SQLAlchemy с фактической схемой БД и генерация diff-скрипта.
- **Ручные операции через `op.execute()`** — выполнение произвольного SQL для конструкций, которые декларативный API SQLAlchemy не выражает.
- **Цепочка ревизий** (`revision` / `down_revision`) — линейная история изменений схемы, хранящаяся в таблице `alembic_version`.
- **Программные операции** (`op.create_table`, `op.create_index`, `op.create_check_constraint`) — типизированное описание DDL.

**Преимущества:** тесная интеграция с SQLAlchemy (миграции пишутся в тех же терминах, что и модели), воспроизводимость схемы на любом окружении, возможность отката, поддержка raw SQL для специфичных для PostgreSQL конструкций.

### 1.3. Контекст Применения в Проекте
**Где использована:** каталог `backend/alembic/versions/` — вся схема БД проекта (10 таблиц) создается и изменяется исключительно через миграции; в Docker-развертывании `alembic upgrade head` выполняется автоматически при старте контейнера бэкенда.

**Решаемые задачи:** ключевая бизнес-гарантия проекта — невозможность двойного бронирования мастера — реализована констрейнтом PostgreSQL `EXCLUDE USING gist`. Эта конструкция **не выражается** через декларативный API SQLAlchemy, поэтому единственное корректное место для нее — миграция с raw SQL. Аналогично через миграции созданы частичные unique-индексы (`WHERE deleted_at IS NULL`), совместимые с мягким удалением записей.

**Обоснование выбора:** Alembic — де-факто единственный стандарт миграций для SQLAlchemy-проектов (аналог `manage.py migrate` в Django). Автогенерация экономит время на рутинных изменениях, а механизм `op.execute()` позволил включить в версионируемую историю и специфичный для PostgreSQL SQL.

### 1.4. Пример Реализации
Фрагмент начальной миграции `backend/alembic/versions/0001_initial.py` — установка расширения и создание exclusion-констрейнта против двойного бронирования:

```python
def upgrade() -> None:
    # Расширение btree_gist позволяет комбинировать в одном GiST-индексе
    # скалярное равенство (master_id) и диапазонный оператор пересечения.
    op.execute("CREATE EXTENSION IF NOT EXISTS btree_gist")

    # ... op.create_table("users", ...), op.create_table("appointments", ...) ...

    # EXCLUDE USING gist — нельзя выразить через SQLAlchemy, пишем SQL напрямую.
    # WHERE (status <> 'cancelled') — отменённая запись не блокирует слот навсегда.
    op.execute("""
        ALTER TABLE appointments
        ADD CONSTRAINT no_double_booking
        EXCLUDE USING gist (
            master_id WITH =,
            tstzrange(start_time, end_time, '[)') WITH &&
        )
        WHERE (status <> 'cancelled')
    """)
```

**Пояснение к коду:**
- **Что делает:** запрещает на уровне СУБД существование двух не-отмененных записей к одному мастеру (`master_id WITH =`) с пересекающимися временными интервалами (`tstzrange(...) WITH &&`); полуоткрытый интервал `'[)'` позволяет следующей записи начинаться ровно в момент окончания предыдущей.
- **Как используется Alembic:** метод `op.execute()` встраивает raw SQL в версионируемую миграцию — конструкция попадает в историю схемы наравне с автогенерируемыми таблицами и воспроизводится на каждом окружении.
- **Какую проблему решает:** race condition при одновременном бронировании — проверка пересечений в коде приложения (`AppointmentRepository.get_overlapping()`) остается лишь предварительным фильтром для понятного сообщения об ошибке, а констрейнт дает атомарную гарантию.

### 1.5. Влияние и Результат
**Влияние на разработку:** схема из 10 таблиц с индексами и констрейнтами воспроизводится одной командой; при доработках (добавление блокировки пользователей, журнала входов) новые миграции генерировались автоматически и накатывались без ручного вмешательства в БД.

**Результат:** двойное бронирование стало физически невозможным — при конкурентной вставке проигравшая транзакция получает `IntegrityError`, который сервисный слой превращает в понятный ответ 409. Docker-развертывание получило автоматическую актуализацию схемы при старте.

**Сложности/Ограничения:** автогенерация «видит» только модели, импортированные в `models/__init__.py` — пропущенный импорт означает молча пропущенную таблицу (в проекте это закреплено правилом: файл обязан импортировать каждую модель). Также выяснилось, что колонка `alembic_version.version_num` ограничена 32 символами — слишком длинный идентификатор ревизии пришлось сокращать.

### 1.6. Альтернативы
- **Django Migrations** — встроены в Django, но проект использует FastAPI + SQLAlchemy, где их нет.
- **Raw SQL-скрипты / yoyo-migrations** — не дают автогенерации по ORM-моделям и сверки схемы; вся рутина ложится на разработчика.

**Почему не выбраны:** Alembic — официальный инструмент экосистемы SQLAlchemy, альтернативы либо привязаны к другому фреймворку, либо беднее функционально.

## 2. python-jose

### 2.1. Название и Тип
**python-jose** (версия 3.5). Тип: библиотека Python, реализация стандартов JOSE (JavaScript Object Signing and Encryption): JWT (RFC 7519), JWS, JWE, JWK. Разработчик: сообщество (Michael Davis, mpdavis), open source.

### 2.2. Цель и Функциональность
**Главная задача:** создание и верификация криптографически подписанных токенов JSON Web Token — отраслевого стандарта stateless-аутентификации, при которой сервер не хранит сессии: вся информация о пользователе и сроке действия зашита в самоподписанный токен.

**Ключевые возможности (использованные в проекте):**
- `jwt.encode(payload, key, algorithm)` — генерация подписанного токена с полезной нагрузкой (идентификатор пользователя `sub`, срок действия `exp`).
- `jwt.decode(token, key, algorithms)` — верификация подписи и **автоматическая проверка срока действия** (`exp`): просроченный токен вызывает исключение без ручных проверок.
- Исключение `JWTError` — единая точка обработки всех невалидных токенов (поддельных, просроченных, поврежденных).

**Преимущества:** полная реализация RFC, автоматическая валидация стандартных claims, простой API из двух функций для типового сценария, совместимость с любым JWT-клиентом.

### 2.3. Контекст Применения в Проекте
**Где использована:** `backend/app/services/auth_service.py` — единственное место в проекте, где создаются и разбираются токены; выданный токен фронтенд затем передает в заголовке `Authorization: Bearer <token>` с каждым запросом.

**Решаемые задачи:** аутентификация всех защищенных эндпоинтов API и фундамент ролевой модели (RBAC): из токена извлекается `user_id`, по нему загружается пользователь, далее зависимость `require_role()` проверяет роль (клиент/мастер/администратор).

**Обоснование выбора:** ТЗ требует токенную аутентификацию с разграничением по ролям. JWT выбран за stateless-природу (не нужна таблица сессий и ее чистка), python-jose — за полноту реализации и статус библиотеки, используемой в официальной документации FastAPI по безопасности, что упростило освоение по готовым материалам.

### 2.4. Пример Реализации
Фрагмент `auth_service.py` — выпуск и верификация токена:

```python
from jose import JWTError, jwt

def _create_access_token(user_id: UUID) -> str:
    expire = datetime.now(timezone.utc) + timedelta(
        minutes=settings.access_token_expire_minutes
    )
    payload = {"sub": str(user_id), "exp": expire}
    return jwt.encode(payload, settings.secret_key, algorithm=settings.algorithm)

def _decode_token(token: str) -> Optional[UUID]:
    try:
        payload = jwt.decode(
            token, settings.secret_key, algorithms=[settings.algorithm]
        )
        user_id = payload.get("sub")
        return UUID(user_id) if user_id else None
    except (JWTError, ValueError):
        return None
```

**Пояснение к коду:**
- **Что делает:** при успешном входе `_create_access_token` формирует токен с UUID пользователя и сроком действия; `_decode_token` на каждом защищенном запросе проверяет подпись и срок, возвращая `None` для любого невалидного токена (далее это превращается в ответ 401).
- **Как используется библиотека:** пара `jwt.encode`/`jwt.decode` с симметричным ключом (`SECRET_KEY` из конфигурации) и алгоритмом HS256; claim `exp` проверяется библиотекой автоматически — просроченный токен выбрасывает `JWTError`. Важная деталь безопасности: `algorithms` при декодировании задан **явным списком**, что исключает атаку подмены алгоритма (`alg: none`).
- **Какую проблему решает:** stateless-сессии без хранения на сервере; токен невозможно подделать или «продлить» без знания секретного ключа.

### 2.5. Влияние и Результат
**Влияние на разработку:** вся аутентификация уложилась в ~30 строк; интеграция с DI-системой FastAPI (`Depends(bearer_scheme)`) дала автоматическое поле авторизации в Swagger UI — тестировать защищенные эндпоинты можно прямо из документации.

**Результат:** работает трехролевая модель доступа; токены имеют ограниченный срок жизни; фронтенд хранит токен в `localStorage` и прозрачно подставляет его через интерсептор Axios.

**Сложности/Ограничения:** JWT принципиально **не отзываем** до истечения срока — для реализации блокировки пользователя администратором пришлось добавить проверку флага `is_blocked` при каждой аутентификации (`get_current_user`), а не полагаться на сам токен. Также python-jose тянет зависимости (`ecdsa`, `rsa`), избыточные для используемого HS256.

### 2.6. Альтернативы
- **PyJWT** — более легковесная библиотека с тем же API; сегодня рекомендуется документацией FastAPI.
- **Серверные сессии (кука + таблица сессий)** — дают мгновенный отзыв, но требуют состояния на сервере и CSRF-защиты.

**Почему не выбраны:** python-jose был выбран по актуальным на момент старта материалам FastAPI и полностью покрывает задачу; миграция на PyJWT возможна малой кровью (совпадающий API), но не дает практической выгоды. Серверные сессии противоречили бы требованию токенной аутентификации REST API из ТЗ.

## 3. bcrypt

### 3.1. Название и Тип
**bcrypt** (версия 5.0). Тип: библиотека Python (биндинг к реализации на C/Rust) алгоритма адаптивного хеширования паролей bcrypt (Нильс Провос, Дэвид Мазьер, 1999). Разработчик: Python Cryptographic Authority (PyCA).

### 3.2. Цель и Функциональность
**Главная задача:** безопасное необратимое хеширование паролей. В отличие от быстрых хешей общего назначения (SHA-256), bcrypt **намеренно медленный** и содержит настраиваемый фактор сложности — перебор миллиардов кандидатов на GPU становится экономически бессмысленным.

**Ключевые возможности (использованные в проекте):**
- `bcrypt.gensalt()` — генерация криптографически случайной соли (соль встраивается в итоговую строку хеша, отдельного хранения не требует).
- `bcrypt.hashpw(password, salt)` — вычисление хеша при регистрации и смене пароля.
- `bcrypt.checkpw(password, hashed)` — проверка пароля при входе за константное время сравнения.

**Преимущества:** проверенный четвертьвековой практикой алгоритм, автоматическое управление солью (радужные таблицы бесполезны), настраиваемая стоимость вычисления, простейший API из трех функций.

### 3.3. Контекст Применения в Проекте
**Где использована:** `backend/app/services/auth_service.py` — регистрация, вход, восстановление пароля. Хеш хранится в колонке `users.password_hash`; исходный пароль не сохраняется и не логируется нигде.

**Решаемые задачи:** защита учетных данных при гипотетической утечке БД; кроме того, bcrypt задействован в защите от **перечисления пользователей** (см. пример ниже) — по времени ответа сервера нельзя определить, зарегистрирован ли email.

**Обоснование выбора:** хранение паролей в открытом виде или через быстрые хеши — прямое нарушение требований безопасности ТЗ (OWASP A02: Cryptographic Failures). Выбран современный биндинг `bcrypt` от PyCA напрямую, без устаревшей обертки `passlib` (не обновлялась с 2020 года и несовместима с bcrypt 4+).

### 3.4. Пример Реализации
Фрагмент `auth_service.py` — хеширование и защита от тайминг-атаки перечисления email:

```python
import bcrypt

def hash_password(password: str) -> str:
    return bcrypt.hashpw(password.encode(), bcrypt.gensalt()).decode()

def _verify_password(password: str, hashed: str) -> bool:
    # bcrypt 5.x бросает ValueError на пароль длиннее 72 байт — такой пароль
    # заведомо неверен (валидные не проходят регистрацию), а не повод для 500.
    if len(password.encode("utf-8")) > 72:
        return False
    return bcrypt.checkpw(password.encode(), hashed.encode())

# Хеш заведомо несуществующего пароля: когда пользователь не найден, bcrypt
# всё равно выполняется — иначе быстрый ответ выдаёт, что email не зарегистрирован.
_DUMMY_HASH = bcrypt.hashpw(secrets.token_bytes(32), bcrypt.gensalt()).decode()

# ...внутри AuthService.login():
user = self.user_repo.get_by_email(email)
hashed = user.password_hash if user else _DUMMY_HASH
password_ok = _verify_password(password, hashed) and user is not None
```

**Пояснение к коду:**
- **Что делает:** при регистрации пароль превращается в хеш со случайной солью; при входе проверка выполняется **всегда**, даже если пользователь с таким email не найден — тогда сравнение идет с заранее вычисленным фиктивным хешем.
- **Как используется библиотека:** тройка `gensalt`/`hashpw`/`checkpw`; учтена особенность bcrypt 5.x — жесткий лимит 72 байта на пароль обрабатывается явно, чтобы аномально длинный ввод давал корректный отказ, а не ошибку 500.
- **Какую проблему решает:** утечка БД не раскрывает пароли; время ответа на неверный вход одинаково для существующих и несуществующих аккаунтов, что закрывает канал перечисления зарегистрированных email.

### 3.5. Влияние и Результат
**Влияние на разработку:** минимальное — API библиотеки тривиален; больше времени заняло проектирование связанной логики (dummy-хеш, лимит попыток входа через журнал `login_attempts`).

**Результат:** пароли хранятся согласно современным практикам; вход защищен и от перебора (rate limiting: 5 неудач за 15 минут), и от перечисления аккаунтов.

**Сложности/Ограничения:** намеренная медленность bcrypt (~100–200 мс) заметно замедлила бы юнит-тесты — тесты проекта используют подмену функций хеширования. Лимит 72 байта потребовал явной обработки после обновления до bcrypt 5.x, где поведение сменилось с тихого усечения на исключение.

### 3.6. Альтернативы
- **Argon2 (argon2-cffi)** — победитель Password Hashing Competition 2015, устойчивее к GPU/ASIC-перебору за счет требований к памяти; технически лучший выбор для новых систем.
- **hashlib.pbkdf2_hmac** — из стандартной библиотеки, без внешних зависимостей, но слабее к GPU-атакам и требует ручного управления солью и числом итераций.

**Почему не выбраны:** bcrypt при корректной стоимости достаточен для масштабов проекта, лучше знаком команде и повсеместно описан в учебных материалах FastAPI; преимущество Argon2 в данном контексте не окупает дополнительную зависимость.

## 4. Pinia

### 4.1. Название и Тип
**Pinia** (версия 2.x). Тип: библиотека управления состоянием (state management) для Vue 3; официальный инструмент экосистемы Vue, преемник Vuex. Разработчик: Эдуардо Сан-Мартин Мороте (Eduardo San Martín Morote, core team Vue), поддерживается командой Vue.

### 4.2. Цель и Функциональность
**Главная задача:** централизованное хранение состояния SPA, разделяемого между компонентами. Без стора данные вроде «текущий пользователь» пришлось бы прокидывать через пропсы через все дерево компонентов или дублировать запросы к API в каждом компоненте.

**Ключевые возможности (использованные в проекте):**
- `defineStore()` в **setup-синтаксисе** — стор описывается теми же примитивами Composition API (`ref`, `computed`), что и компоненты, без отдельного словаря mutations.
- Реактивные геттеры (`computed`) — производные значения (`isLoggedIn`, `isAdmin`) автоматически обновляют все использующие их компоненты.
- Асинхронные действия — методы стора свободно используют `async/await` для запросов к API.
- Модульность — независимые сторы по доменам: `auth` (сессия), `toast` (глобальные уведомления), `masterProfile` (кеш профиля мастера).

**Преимущества:** официальный статус и полная интеграция с Vue DevTools, минимальный бойлерплейт по сравнению с Vuex (нет mutations), tree-shaking неиспользуемых сторов, естественная типизация.

### 4.3. Контекст Применения в Проекте
**Где использована:** `frontend/src/stores/` — три стора; ключевой — `auth.js`, единственный источник истины о сессии пользователя для всего приложения (шапка сайта, маршрутизатор, страницы кабинетов).

**Решаемые задачи:** роль пользователя определяет всё поведение интерфейса — навигацию, доступные страницы (`/dashboard` для мастера, `/admin` для администратора), защиту маршрутов. Стор `auth` хранит токен и профиль, а флаг `ready` решает конкретную производственную проблему: глобальный охранник маршрутизатора обязан дождаться первичной загрузки профиля (`fetchMe()`), иначе при перезагрузке страницы залогиненного пользователя выбрасывало бы на `/login`.

**Обоснование выбора:** Pinia — официальная рекомендация Vue 3 (Vuex переведен в режим поддержки); setup-синтаксис сторов идентичен стилю `<script setup>` компонентов проекта, что снижает порог входа.

### 4.4. Пример Реализации
Фрагмент `frontend/src/stores/auth.js`:

```javascript
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import { authApi } from '../api'

export const useAuthStore = defineStore('auth', () => {
  const token = ref(localStorage.getItem('token') || null)
  const user = ref(null)
  const ready = ref(false)   // роутер ждёт этот флаг перед проверкой прав

  const isLoggedIn = computed(() => !!token.value)
  const isAdmin = computed(() => user.value?.role === 'admin')

  async function fetchMe() {
    if (!token.value) { ready.value = true; return }
    try {
      const res = await authApi.me()
      user.value = res.data
    } catch (err) {
      // Разлогиниваем только если токен действительно отвергнут (401).
      // Сетевая ошибка или 500 не повод стирать валидную сессию.
      if (err.response?.status === 401) logout()
    } finally {
      ready.value = true
    }
  }

  function logout() {
    token.value = null
    user.value = null
    localStorage.removeItem('token')
  }

  return { token, user, ready, isLoggedIn, isAdmin, fetchMe, logout }
})
```

**Пояснение к коду:**
- **Что делает:** восстанавливает сессию из `localStorage` при старте приложения, загружает профиль с сервера и предоставляет всем компонентам реактивные признаки роли.
- **Как используется библиотека:** `defineStore` с setup-функцией; `ref` для состояния, `computed` для производных значений — любой компонент, вызвавший `useAuthStore()`, автоматически перерисуется при изменении сессии.
- **Какую проблему решает:** единая точка правды о сессии; аккуратная обработка ошибок (`fetchMe` различает «токен отвергнут» и «сеть недоступна») предотвращает ложные разлогины при временных сбоях.

### 4.5. Влияние и Результат
**Влияние на разработку:** исчезла необходимость прокидывать пользователя через пропсы; охранники маршрутов, шапка и страницы читают одно состояние. Vue DevTools показывает содержимое сторов, что ускорило отладку сценариев входа/выхода.

**Результат:** корректная работа ролевой навигации, переживание перезагрузки страницы без потери сессии, глобальные toast-уведомления из любого места приложения через отдельный стор.

**Сложности/Ограничения:** порядок инициализации имеет значение — стор нельзя вызвать до `app.use(pinia)`, из-за чего обработчик 401 в HTTP-клиенте пришлось внедрять из `main.js`, а не импортировать стор напрямую (иначе возникал циклический импорт `client → router → stores → client`).

### 4.6. Альтернативы
- **Vuex 4** — предшественник; требует бойлерплейта (mutations), официально в режиме поддержки без развития.
- **provide/inject + composables** — встроенные средства Vue; достаточны для мелких приложений, но не дают DevTools-инспекции и единой структуры сторов.

**Почему не выбраны:** Pinia — официальная рекомендация именно для Vue 3, синтаксически едина с Composition API проекта и не имеет недостатков Vuex.

## 5. Axios

### 5.1. Название и Тип
**Axios** (версия 1.x). Тип: HTTP-клиент для браузера и Node.js, библиотека JavaScript на основе Promise. Разработчик: сообщество open source (Matt Zabriskie и контрибьюторы), проект спонсируется через OpenCollective.

### 5.2. Цель и Функциональность
**Главная задача:** единообразное выполнение HTTP-запросов с расширяемой обработкой: базовые URL, заголовки по умолчанию, перехватчики (interceptors) запросов и ответов, автоматическая (де)сериализация JSON, нормальная обработка ошибок по статус-кодам.

**Ключевые возможности (использованные в проекте):**
- `axios.create(config)` — изолированный экземпляр клиента с `baseURL: '/api'` и заголовками по умолчанию.
- **Перехватчик запросов** — автоматическая подстановка JWT в заголовок `Authorization` каждого запроса.
- **Перехватчик ответов** — централизованная реакция на 401 (сброс сессии и редирект на страницу входа) для всех запросов приложения разом.
- Отклонение промиса при статусах 4xx/5xx — ошибки API обрабатываются через `catch` с доступом к `error.response.status` и телу ответа.

**Преимущества:** перехватчики (главное отличие от `fetch`), меньше рутины при работе с JSON, единая точка конфигурации, автоматическая защита от XSRF.

### 5.3. Контекст Применения в Проекте
**Где использована:** `frontend/src/api/client.js` — единственный экземпляр клиента; каждый ресурс API оформлен отдельным модулем (`auth.js`, `masters.js`, `appointments.js` и т.д.), реэкспортируемым из `src/api/index.js`. Ни один компонент не вызывает axios напрямую.

**Решаемые задачи:** все взаимодействие SPA с REST API бэкенда — от каталога услуг до админ-панели; сквозная аутентификация (~40 защищенных эндпоинтов) без единого повторения кода подстановки токена; единообразная обработка протухшей сессии.

**Обоснование выбора:** решающим фактором стали перехватчики: подстановка `Bearer`-токена и глобальная обработка 401 задаются один раз. Нативный `fetch` потребовал бы собственную обертку с тем же функционалом — то есть написание «маленького axios» вручную.

### 5.4. Пример Реализации
Файл `frontend/src/api/client.js` целиком:

```javascript
import axios from 'axios'

const client = axios.create({
  baseURL: '/api',
  headers: { 'Content-Type': 'application/json' },
})

// Перехватчик запросов: JWT подставляется в каждый запрос автоматически
client.interceptors.request.use((config) => {
  const token = localStorage.getItem('token')
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

// Обработчик 401 внедряется из main.js (setUnauthorizedHandler): прямой
// импорт router отсюда создавал цикл client → router → stores → client,
// а чистить надо не только localStorage, но и Pinia-стор (auth.logout()).
let onUnauthorized = null

export function setUnauthorizedHandler(handler) {
  onUnauthorized = handler
}

client.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.response?.status === 401 && onUnauthorized) {
      onUnauthorized()
    }
    return Promise.reject(error)
  }
)

export default client
```

**Пояснение к коду:**
- **Что делает:** создает единый HTTP-клиент приложения; каждый исходящий запрос получает актуальный токен, каждый ответ 401 (просроченный/отозванный токен, заблокированный аккаунт) приводит к сбросу сессии и редиректу на вход.
- **Как используется библиотека:** `axios.create` для изоляции конфигурации, `interceptors.request` и `interceptors.response` — ключевые механизмы; ошибка пробрасывается дальше (`Promise.reject`), чтобы вызывающий код мог показать свое сообщение.
- **Какую проблему решает:** исключает дублирование логики аутентификации в десятках вызовов API и гарантирует, что «тихая» смерть сессии в любом месте приложения обработается одинаково. Внедрение обработчика извне (вместо импорта роутера) решает реальную проблему циклического импорта модулей.

### 5.5. Влияние и Результат
**Влияние на разработку:** модули API свелись к однострочным функциям вида `login: (data) => client.post('/auth/login', data)`; вся сквозная логика живет в одном файле на 33 строки.

**Результат:** предсказуемое поведение при истечении сессии на любой странице; в паре с dev-прокси Vite (`/api` → `localhost:8000`) фронтенд в разработке работает без CORS-настроек.

**Сложности/Ограничения:** перехватчик 401 срабатывает и на неудачную попытку входа (что корректно — обработчик проверяет контекст), а взаимосвязь клиента с Pinia-стором потребовала описанной инверсии зависимости. Размер библиотеки (~30 КБ) для SPA несущественен.

### 5.6. Альтернативы
- **Fetch API** — встроен в браузер, ноль зависимостей; но без перехватчиков, `baseURL` и отклонения промисов на 4xx/5xx — все это пришлось бы реализовать самостоятельно.
- **ky / ofetch** — легковесные обертки над fetch с похожими возможностями.

**Почему не выбраны:** axios — самое распространенное решение с наибольшим объемом документации и примеров; экономия в размере от альтернатив не окупает потерю зрелых перехватчиков и привычного API.

## 6. Tailwind CSS

### 6.1. Название и Тип
**Tailwind CSS** (версия 3.4). Тип: utility-first CSS-фреймворк; подключается как плагин PostCSS в сборку Vite. Разработчик: Tailwind Labs (Адам Уотан / Adam Wathan).

### 6.2. Цель и Функциональность
**Главная задача:** стилизация интерфейса композицией атомарных утилитарных классов (`flex`, `px-4`, `text-sm`) прямо в разметке — вместо написания отдельных CSS-файлов с придумыванием имен классов. На этапе сборки сканер удаляет все неиспользуемые утилиты, итоговый CSS содержит только реально примененные правила.

**Ключевые возможности (использованные в проекте):**
- **Дизайн-токены через `theme.extend`** — собственная палитра бренда, шрифты и радиусы задаются один раз в `tailwind.config.js` и становятся классами (`bg-brand-900`, `font-display`).
- Утилиты компоновки (flexbox/grid), отступов, типографики — верстка всех страниц.
- Модификаторы состояний (`hover:`, `focus:`, `disabled:`) — интерактивные состояния без ручного CSS.
- Адаптивные префиксы (`md:`, `lg:`) — брейкпоинты в разметке.
- **Purge/JIT по `content`** — в продакшен-бандл попадают только классы, встреченные в `.vue`-файлах.

**Преимущества:** скорость верстки (нет переключения между разметкой и стилями), невозможность «мертвого CSS», единообразие через ограниченную шкалу значений, крошечный итоговый бандл.

### 6.3. Контекст Применения в Проекте
**Где использована:** все `.vue`-компоненты фронтенда — от публичного каталога до админ-панели; конфигурация — `frontend/tailwind.config.js`; библиотека переиспользуемых примитивов `components/ui/` (BaseButton, BaseCard, StatusPill и др.) построена на утилитах Tailwind.

**Решаемые задачи:** реализация фирменного стиля барбершопа (дизайн-система описана в `design-system/saitama-barbershop/MASTER.md`): монохромная палитра с янтарным акцентом, шрифт Golos Text, нарочито жесткие прямые углы. Ключевой прием — **переопределение всей шкалы `borderRadius` на 1–2 пикселя**, что придает каждому элементу «резкий» характер бренда без слежки за каждым компонентом.

**Обоснование выбора:** компонентные UI-библиотеки (Vuetify, Element Plus) навязывают чужой визуальный язык (Material Design), который пришлось бы перебарывать; чистый CSS/SCSS медленнее в итерациях и накапливает мертвые стили. Tailwind дал полный контроль над дизайном при высокой скорости верстки.

### 6.4. Пример Реализации
Фрагмент `frontend/tailwind.config.js` — дизайн-токены бренда:

```javascript
/** @type {import('tailwindcss').Config} */
export default {
  content: ['./index.html', './src/**/*.{vue,js,ts,jsx,tsx}'],
  theme: {
    extend: {
      fontFamily: {
        display: ['"Golos Text"', 'sans-serif'],
        sans: ['"Golos Text"', 'sans-serif'],
      },
      colors: {
        brand:  { 900: '#111111', 800: '#3A3A3A', 700: '#5C5C5C' },
        accent: { 400: '#FBBF24', 100: '#ECEAE5' },  // янтарный акцент
        stone:  { 50: '#F4F3F0', 200: '#E3E1DC' },
        danger: '#B91C1C',
        success: '#15803D',
      },
      // Фирменный приём: весь UI с почти прямыми углами —
      // переопределяем всю шкалу радиусов, а не правим каждый компонент.
      borderRadius: {
        DEFAULT: '1px', sm: '1px', md: '2px',
        lg: '2px', xl: '2px', '2xl': '2px', '3xl': '2px',
      },
    },
  },
  plugins: [],
}
```

И применение токенов в переиспользуемом компоненте кнопки (`components/ui/BaseButton.vue`) — варианты и размеры собраны из утилит Tailwind:

```html
<template>
  <button
    :type="type"
    :disabled="disabled || loading"
    class="inline-flex items-center justify-center gap-2 whitespace-nowrap
           border font-mono uppercase tracking-wider transition-colors
           duration-200 cursor-pointer disabled:cursor-not-allowed
           disabled:opacity-50"
    :class="[variants[variant] ?? variants.primary, sizes[size] ?? sizes.md]"
  >
    <slot />
  </button>
</template>

<script setup>
const variants = {
  primary: 'bg-brand-900 text-white border-brand-900 hover:bg-brand-800 hover:border-brand-800',
  ghost:   'bg-transparent text-brand-900 border-brand-900 hover:bg-brand-900 hover:text-white',
  danger:  'bg-danger text-white border-danger hover:brightness-90',
}
const sizes = {
  sm: 'px-3 py-1.5 text-[11px]',
  md: 'px-5 py-3 text-xs',
  lg: 'px-8 py-4 text-sm',
}
</script>
```

**Пояснение к коду:**
- **Что делает:** конфигурация объявляет палитру, шрифты и радиусы бренда; компонент кнопки собирает внешний вид из этих токенов — варианты (`primary`/`ghost`/`danger`) и размеры описаны словарями утилитарных классов, включая состояния наведения и неактивности (`disabled:opacity-50`).
- **Как используется библиотека:** механизм `theme.extend` превращает дизайн-решения в систему классов; массив `content` обеспечивает попадание в бандл только использованных утилит.
- **Какую проблему решает:** единообразие стиля во всем приложении — смена значения токена в конфиге мгновенно меняет все страницы; «резкие углы» бренда применяются глобально одним блоком.

### 6.5. Влияние и Результат
**Влияние на разработку:** верстка страниц значительно ускорилась — стили пишутся в том же файле, что и шаблон компонента; дизайн-токены исключили расползание оттенков и отступов между страницами.

**Результат:** консистентный фирменный интерфейс на ~30 страницах/компонентах; продакшен-CSS после purge занимает единицы килобайт; библиотека `components/ui/` позволяет собирать новые страницы из готовых примитивов.

**Сложности/Ограничения:** длинные строки классов ухудшают читаемость сложных компонентов — это смягчено выносом повторяющихся паттернов в компоненты (`BaseButton`, `BaseCard`), а не копированием классов. Purge-сканер не видит динамически склеенных имен классов (`'bg-' + color`) — такие места требуют полных имен классов в коде.

### 6.6. Альтернативы
- **Vuetify / Element Plus** — готовые компонентные библиотеки для Vue: быстрый старт, но узнаваемый чужой дизайн и сложность глубокой кастомизации под фирменный стиль.
- **SCSS с БЭМ-методологией** — полный контроль без зависимостей, но медленнее в итерациях, требует дисциплины именования и накапливает неиспользуемые стили.

**Почему не выбраны:** проекту требовался самобытный визуальный стиль (дизайн-система прорабатывалась отдельно), что исключило компонентные библиотеки; Tailwind сохранил скорость разработки, которой не дал бы ручной SCSS.

## Заключение

Все шесть инструментов объединяет общий принцип отбора: каждый закрывает конкретную производственную задачу проекта штатным, отраслево-стандартным способом и интегрируется с основным стеком без «клея». Alembic сделал воспроизводимой схему БД вместе с ключевым exclusion-констрейнтом против двойного бронирования; python-jose и bcrypt реализовали токенную аутентификацию и безопасное хранение паролей согласно требованиям ТЗ; Pinia, Axios и Tailwind CSS обеспечили SPA-фронтенду единое состояние сессии, централизованное HTTP-взаимодействие и консистентный фирменный интерфейс. Отмеченные ограничения (неотзываемость JWT, лимит bcrypt в 72 байта, циклические импорты при связке Axios и Pinia) были выявлены и обработаны в коде проекта явно.
