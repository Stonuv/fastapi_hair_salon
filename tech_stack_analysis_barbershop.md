# Сравнительный анализ технологических стеков для разработки CMS онлайн-записи барбершопа «Сайтама»

---

## 1. Введение

### Описание проекта
В рамках данного анализа рассматривается выбор оптимального технологического стека для разработки **веб-системы онлайн-записи барбершопа «Сайтама»** — CMS, объединяющей публичный сайт-каталог (услуги, мастера, отзывы), клиентский кабинет с онлайн-бронированием, рабочее место мастера и административную панель. Основная цель системы — автоматизация записи клиентов с гарантированным исключением двойного бронирования одного мастера на пересекающиеся интервалы времени.

### Основные функциональные требования
1. **Онлайн-бронирование с контролем пересечений:** Клиент выбирает мастера, услугу и свободный слот; система обязана атомарно исключать запись двух клиентов к одному мастеру на пересекающееся время — в том числе при конкурентных запросах (race condition).
2. **Ролевая модель (RBAC):** Три роли — клиент, мастер, администратор — с разграничением доступа на уровне API (JWT-аутентификация).
3. **Управление расписанием:** Индивидуальный график работы каждого мастера по дням недели; генерация доступных слотов с учетом длительности услуги и уже существующих записей.
4. **Жизненный цикл записи:** Конечный автомат статусов (`pending → confirmed → done`, отмена из нетерминальных состояний) с валидацией переходов.
5. **Отзывы и рейтинги:** Отзыв привязывается к завершенной записи; агрегация рейтингов мастеров и услуг.
6. **Административная панель:** CRUD по пользователям, мастерам, услугам; статистика (выручка, загрузка мастеров); блокировка пользователей и мягкое удаление (soft delete).
7. **Поиск, фильтрация, пагинация** на всех списочных эндпоинтах.

### Нефункциональные требования
* **Целостность данных:** Двойное бронирование недопустимо ни при каких условиях — защита должна существовать на уровне СУБД, а не только приложения.
* **Безопасность:** Хеширование паролей, защита от перебора при входе (rate limiting), устойчивость к OWASP Top 10, безопасное восстановление пароля.
* **Автодокументируемость API:** Интерактивная документация (Swagger/OpenAPI) — обязательное требование ТЗ.
* **Строгая типизация и валидация:** Валидация всех входных данных на границе API; типизированные модели данных.
* **Простота поддержки:** Проект учебный/портфолио — низкий порог входа и читаемая слоистая архитектура важнее пиковой производительности.

## 2. Альтернативные стеки

Для детального анализа выбраны три технологических стека, представляющие различные архитектурные философии и реалистичные для небольшой команды (1–2 разработчика) в рамках учебного проекта.

### Стек 1: Python (FastAPI) + Vue 3 (SPA) + PostgreSQL
* **Бэкенд:** `FastAPI` — современный асинхронный Python-фреймворк на базе Starlette и Pydantic. Автоматическая генерация OpenAPI-документации, встроенная система зависимостей (`Depends`) для аутентификации и авторизации, валидация запросов/ответов «из коробки».
* **ORM/миграции:** `SQLAlchemy 2.0` (типизированный `Mapped[...]`-стиль) + `Alembic`.
* **Фронтенд:** `Vue 3` (`<script setup>`) + `Pinia` + `Vite` + `Tailwind CSS` — легковесный SPA-фреймворк с низким порогом входа и официальной русскоязычной документацией.
* **СУБД:** `PostgreSQL` — поддерживает `EXCLUDE USING gist`-констрейнты по временным диапазонам (`tstzrange`), что даёт гарантию отсутствия пересекающихся броней на уровне самой базы данных.

### Стек 2: Node.js (NestJS) + React (SPA) + MongoDB
* **Бэкенд:** `NestJS` — TypeScript-фреймворк с модульной архитектурой (в духе Angular/Spring), декораторами и DI-контейнером.
* **Фронтенд:** `React` — самая распространенная SPA-библиотека с крупнейшей экосистемой компонентов.
* **СУБД:** `MongoDB` — документоориентированная NoSQL СУБД с гибкой схемой; транзакции поддерживаются, но декларативных констрейнтов уровня SQL (уникальность по условию, exclusion constraints) нет.

### Стек 3: PHP (Laravel) + Blade/Livewire + MySQL
* **Бэкенд + фронтенд:** `Laravel` — «всё включено»: ORM Eloquent, миграции, очереди, аутентификация, шаблонизатор Blade и `Livewire` для интерактивности без отдельного SPA.
* **СУБД:** `MySQL` — зрелая реляционная СУБД с полной поддержкой транзакций, но без exclusion-констрейнтов; защита от двойного бронирования реализуема только блокировками (`SELECT ... FOR UPDATE`) в коде приложения.

## 4. Оценка стеков по критериям (Аргументированная)

### 1. Гарантии целостности бронирования (Вес: 3)
* **Стек 1 (FastAPI + PostgreSQL): 10/10** PostgreSQL — единственная из рассматриваемых СУБД с декларативным `EXCLUDE USING gist`-констрейнтом: одно ограничение на `(master_id, tstzrange(start_time, end_time))` физически запрещает пересекающиеся брони даже при одновременных вставках из параллельных транзакций. Проверка в коде приложения остается лишь для дружелюбного сообщения об ошибке, а не как единственная линия обороны.
* **Стек 2 (NestJS + MongoDB): 4/10** MongoDB поддерживает многодокументные транзакции, но не имеет exclusion-констрейнтов и `CHECK`-ограничений по диапазонам. Защита от гонок реализуема только вручную (оптимистичные блокировки, паттерн «резервирование слота отдельным документом с уникальным индексом»), что хрупко и требует дискретной сетки слотов вместо произвольных интервалов.
* **Стек 3 (Laravel + MySQL): 6/10** MySQL транзакционен и надежен, но exclusion-констрейнтов не имеет: пересечения приходится исключать через `SELECT ... FOR UPDATE` с блокировкой строк мастера. Работает, однако корректность зависит от дисциплины программиста в каждой точке записи, а не от схемы БД.

### 2. Автогенерация API-документации (Вес: 2)
* **Стек 1: 10/10** OpenAPI-схема и интерактивный Swagger UI генерируются FastAPI автоматически из тех же Pydantic-моделей и сигнатур эндпоинтов, которые выполняют валидацию — документация не может «разъехаться» с кодом и не требует ни строчки дополнительной разметки.
* **Стек 2: 8/10** `@nestjs/swagger` даёт качественную документацию, но требует ручного навешивания декораторов (`@ApiProperty`, `@ApiResponse`) на DTO и контроллеры; пропущенный декоратор — пробел в документации.
* **Стек 3: 4/10** В Laravel штатной генерации OpenAPI нет; сторонние пакеты (Scribe, l5-swagger) работают через PHPDoc-аннотации, которые дублируют код и легко устаревают.

### 3. Строгая типизация и валидация (Вес: 2)
* **Стек 1: 10/10** Pydantic v2 валидирует каждый запрос по аннотациям типов (включая `Annotated`-ограничения: длины строк, положительные суммы, timezone-aware даты), а SQLAlchemy 2.0 в `Mapped[...]`-стиле типизирует слой данных. Ошибка валидации автоматически превращается в структурированный ответ 422.
* **Стек 2: 8/10** TypeScript + `class-validator` дают сильную типизацию, но валидация «включается» декораторами на каждом поле DTO, а типы TS стираются в рантайме — расхождение декораторов и типов не ловится компилятором.
* **Стек 3: 5/10** PHP 8 имеет типы, но валидация запросов в Laravel — это массивы строковых правил (`'price' => 'required|numeric|min:0'`) в FormRequest: ошибки в правилах обнаруживаются только в рантайме, статический анализ им не помогает.

### 4. Стабильность и зрелость экосистемы (Вес: 2)
* **Стек 1: 8/10** Python, PostgreSQL и SQLAlchemy — многолетние индустриальные стандарты. FastAPI моложе Django, но API стабилен, а фундамент (Starlette, Pydantic) зрел; Vue 3 давно является основной версией фреймворка.
* **Стек 2: 7/10** NestJS и React зрелы, но экосистема npm печально известна скоростью устаревания пакетов и глубиной дерева зависимостей — стоимость поддержки со временем растет.
* **Стек 3: 9/10** Laravel — эталон предсказуемости: ежегодные мажорные релизы с понятными гайдами миграции, LTS-политика, минимум внешних зависимостей благодаря философии «всё включено».

### 5. Рынок труда в РФ (Вес: 2)
* **Стек 1: 9/10** Python — самый массовый язык обучения в вузах и онлайн-школах РФ; FastAPI в вакансиях уверенно вытесняет Django для новых API-сервисов. Vue традиционно популярен в русскоязычном сегменте.
* **Стек 2: 9/10** JS/TS-разработчики доступны в избытке; React — самый частый фронтенд-запрос в вакансиях.
* **Стек 3: 8/10** PHP/Laravel-разработчиков много, особенно в регионах и веб-студиях, однако доля PHP в новых проектах и его привлекательность для портфолио снижаются.

### 6. Скорость разработки (Вес: 2)
* **Стек 1: 8/10** FastAPI устраняет основной бойлерплейт API (валидация, сериализация, документация, DI для авторизации), Alembic автогенерирует миграции. Минус — админ-панель не встроена, её пришлось разработать как SPA-раздел вручную.
* **Стек 2: 7/10** Единый язык на фронте и бэке ускоряет переключение контекста, но NestJS требует заметного объема церемоний (модули, провайдеры, DTO, декораторы) для каждой сущности.
* **Стек 3: 9/10** Laravel — лидер прототипирования: генераторы `artisan`, Eloquent, готовая аутентификация и пакеты типа Filament для админки позволяют собрать CRUD-ядро за дни. Однако требуемый ТЗ SPA-фронтенд нивелирует часть преимуществ Blade.

### 7. Производительность рантайма (Вес: 1)
* **Стек 1: 8/10** ASGI/uvicorn с асинхронной обработкой отлично держит I/O-нагрузку записи/чтения; для одного салона запас многократный. Чистая вычислительная скорость Python ниже конкурентов, но в данном профиле нагрузки не является узким местом.
* **Стек 2: 8/10** Событийный цикл Node.js эффективен на I/O-bound задачах; сопоставим со Стеком 1.
* **Стек 3: 6/10** Классическая модель PHP «процесс на запрос» медленнее событийных моделей, хотя с OPcache для салонной нагрузки достаточна.

### 8. Безопасность «из коробки» (Вес: 2)
* **Стек 1: 8/10** Экосистема даёт проверенные примитивы: `bcrypt` для паролей, JWT-библиотеки, параметризованные запросы SQLAlchemy исключают SQL-инъекции, Pydantic отсекает неожиданные поля. Rate limiting входа и хеширование токенов сброса пароля реализуются несложным собственным кодом.
* **Стек 2: 7/10** Guards и Pipes NestJS — хорошие абстракции, но базовая гигиена (helmet, CORS, лимиты) собирается из сторонних пакетов вручную, а качество npm-зависимостей требует постоянного аудита.
* **Стек 3: 9/10** Laravel включает защиту от CSRF, XSS (экранирование Blade), SQL-инъекций (Eloquent), хеширование паролей, rate limiting и восстановление пароля штатно — минимум места для архитектурных ошибок.

### 9. Русскоязычная документация (Вес: 2)
* **Стек 1: 9/10** Документация FastAPI официально переведена на русский; Vue имеет полный официальный русский перевод; по Python и PostgreSQL — огромный корпус русскоязычных книг, курсов и статей на Хабре.
* **Стек 2: 7/10** По React и TypeScript качественных русскоязычных материалов много, по NestJS — заметно меньше, документация фреймворка только на английском.
* **Стек 3: 8/10** Laravel исторически хорошо покрыт русскоязычными переводами документации и сообществами (Laravel Framework Russian Community).

## 6. Вывод

На основании проведенного многокритериального анализа для разработки CMS онлайн-записи барбершопа «Сайтама» **рекомендуется выбрать Стек 1 (Python / FastAPI + Vue 3 SPA + PostgreSQL)**.

### Ключевые преимущества выбора:
1. **Максимальное соответствие ядру системы:** Констрейнт `EXCLUDE USING gist` PostgreSQL делает двойное бронирование физически невозможным на уровне схемы данных — прикладная проверка пересечений остается лишь предварительным фильтром для понятного сообщения клиенту. Это единственное решение среди рассмотренных, устойчивое к гонкам без ручных блокировок.
2. **Требования ТЗ закрываются штатно:** Интерактивная Swagger-документация, строгая валидация Pydantic v2, типизированный доступ к данным SQLAlchemy 2.0 и миграции Alembic не требуют дополнительного кода и не могут разойтись с реализацией.
3. **Доступность ресурсов и перспективность:** Python и Vue широко представлены в образовательной среде и на рынке труда РФ, обе технологии имеют официальную русскоязычную документацию, а FastAPI — один из самых востребованных бэкенд-фреймворков для новых проектов, что повышает ценность работы как портфолио.

Выбранный стек оптимально балансирует между гарантиями целостности данных (критичными для домена бронирования), скоростью разработки в рамках учебного семестра и доступностью кадров и учебных материалов.
