**Договорённости, зафиксированные перед началом работы над документом:**
- **Одна инсталляция = одна сеть.** Это не мультитенантный SaaS на несколько независимых сетей-клиентов одновременно. Новый клиент (другая сеть парикмахерских) получает отдельное развёртывание (свой сервер, свой домен, своя БД). Внутри одной инсталляции поддерживается произвольное число точек («салонов») одной сети. Возможный путь к полноценному SaaS в будущем — см. §4.11.
- **Порядок работ**: сначала закрываются продакшен-риски (§3), затем строится функционал сети (§4) — на устойчивом фундаменте.
- **Раздел §4 — полная техническая спецификация**, а не только описание идеи: имена таблиц/колонок/эндпоинтов/ролей финальные, миграции расписаны по шагам, с неё можно начинать разработку напрямую.
- **Онбординг новых точек** — самостоятельный, через панель владельца, без участия разработчика/оператора.

## 1. Текущее состояние проекта

Проект — CMS для барбершопа («Барбершоп «Сайтама»») на FastAPI + SQLAlchemy 2.0 + PostgreSQL 16 (бэкенд) и Vue 3 + Pinia + Vite (фронтенд), изначально учебный (граded coursework против русскоязычной SRS-спеки), но уже доведён до состояния, близкого к реальной эксплуатации: Docker Compose стек (Postgres + backend + frontend/nginx + Caddy + автобэкапы), CI на GitHub Actions, IP-рейтлимиты, полноценная сессионная модель аутентификации, антидвойное бронирование на уровне БД. Судя по истории коммитов, последние ~15 изменений — методичный проход по чек-листу продакшен-готовности (HTTPS → email → security-фиксы → сессии → напоминания → бэкапы → рейтлимиты → Sentry → CI).

**Уже сделано хорошо (контекст, не трогать без причины):**
- Зависимости бэкенда закреплены точными версиями (`backend/requirements.txt`, `==`).
- Alembic-миграции (12 линейных ревизий, `0001`–`0012`), автонакат при старте контейнера (`backend/Dockerfile:14`).
- Защита от двойного бронирования на уровне БД — `EXCLUDE USING gist` (`0001_initial.py`), а не только в коде приложения.
- Автоматические бэкапы Postgres с ретеншном (`prodrigestivill/postgres-backup-local`), подтверждённо работают — реальные датированные архивы лежат в `backups/`.
- Caddy с автоматическим HTTPS (Let's Encrypt) при наличии домена.
- Fail-fast валидация конфигурации: приложение отказывается стартовать вне `DEBUG` с дефолтным `SECRET_KEY` или без `SETUP_TOKEN` (`backend/app/config.py:136-157`).
- JWT в httpOnly-cookie (не localStorage) с реальным отзывом через `token_version` + отдельная сессионная система для refresh-токенов (ротация, привязка к устройству).
- IP-рейтлимиты на регистрацию/бронирование/отзывы + отдельная защита от брутфорса логина (лок по email после 5 неудачных попыток).
- CI: юнит-тесты бэкенда, сборка фронтенда, валидация `docker-compose.yml` (`.github/workflows/ci.yml`).
- Soft-delete с частичными уникальными индексами (`WHERE deleted_at IS NULL`) — освобождает email/телефон при удалении аккаунта.
- Строгая слоистая архитектура (`routes → services → repositories → models`), последовательно применённая по всей кодовой базе.
- `frontend/dist/` (сборка) корректно в `.gitignore`, не замусоривает репозиторий.

Именно на этом фундаменте строится всё дальнейшее — раздел §2 указывает, чего не хватает, §3 и §4 — как это исправить и развить.

## 2. Недостатки проекта (полный аудит)

### 2.1. Критические — блокеры реального запуска на клиентов

**#1. Утечка секрета в истории git.**
`backend/.env` был закоммичен на раннем этапе (`697fd3a`, `8b97a98`, `e4999e4`, а также затронут в `7dd73ec "fixing auth"`) и содержит реальный `DATABASE_URL` с паролем и `SECRET_KEY=nikita_nikita_nikita`. Эти коммиты — подтверждённые предки текущего `HEAD` (`git merge-base --is-ancestor 697fd3a HEAD` → true), то есть содержимое навсегда доступно в истории репозитория, даже несмотря на то что текущие `.env`/`backend/.env` сейчас корректно в `.gitignore`. Похоже на локальные dev-креды, а не боевой секрет, но у репозитория есть реальный GitHub remote — риск реален, если репозиторий публичный или станет публичным.

**#2. Юридически недействующая политика конфиденциальности.**
`frontend/src/views/PrivacyPolicyPage.vue:20-31` — раздел 2 («Оператор персональных данных») буквально содержит плейсхолдеры `[ЗАПОЛНИТЬ ПЕРЕД ПУБЛИКАЦИЕЙ]`/`[ЗАПОЛНИТЬ]` вместо наименования ИП/ООО, ИНН, ОГРН, юридического адреса. В самом коде прокомментировано: без реальных реквизитов документ не имеет юридической силы по 152-ФЗ «О персональных данных». Это жёсткий блокер именно для публичного запуска или продажи доступа реальным клиентам (сбор email/телефона/истории записей без действующей политики — прямое нарушение).

**#3. Мониторинг ошибок подключен, но не активен.**
`backend/app/main.py`/`frontend/src/main.js` инициализируют Sentry только при наличии DSN; `sentry_dsn` по умолчанию `None`, реальный DSN нигде не задан (открытый пункт `new_issues.md` №41 — регистрация на sentry.io была недоступна на момент подключения). Сейчас в продакшене нет вообще никакого автоматического уведомления об ошибках — сбои увидит только тот, кто сам зайдёт смотреть логи контейнера.

**#4. Полное отсутствие security-заголовков.**
Проверены целиком `Caddyfile` и `frontend/nginx.conf` — ни в одном нет ни `Strict-Transport-Security` (HSTS), ни `Content-Security-Policy`, ни `X-Content-Type-Options`, ни `X-Frame-Options`/`frame-ancestors`, ни `Referrer-Policy`. Ни Caddy, ни nginx не переопределяют эти заголовки — сайт полагается только на HTTPS-терминацию Caddy и ничего больше.

### 2.2. Важные

**#5. Нет интеграционных тестов с реальной БД.**
`backend/tests/conftest.py` прямо документирует подход: сервисы тестируются с фейковыми репозиториями (`SimpleNamespace`), без обращения к Postgres. Самая критичная гарантия системы — constraint `EXCLUDE USING gist`, защищающий от двойного бронирования — не покрыта автотестами вообще, только «проверено вручную на стенде» (комментарий в том же файле).

**#6. Нет CD — деплой полностью ручной.**
`git pull && docker compose up --build -d` на VPS вручную (задокументировано в README). CI (тесты/сборка/валидация compose) есть, автоматического деплоя после мерджа — нет.

**#7. Тестовый фреймворк едет в продакшен-образ.**
`pytest==9.1.1` — прямая строка в основном `backend/requirements.txt` (не в отдельном dev/test-файле), `backend/Dockerfile` ставит его в прод-образ через `pip install -r requirements.txt`. Не уязвимость, но лишний вес и поверхность образа.

**#8. Нет healthcheck и непривилегированных пользователей в контейнерах.**
В `docker-compose.yml` healthcheck есть только у `db` (`pg_isready`). У `backend`, `frontend`, `caddy` — нет. Ни в `backend/Dockerfile`, ни в `frontend/Dockerfile` нет директивы `USER` — оба контейнера работают от root.

**#9. Ограничения single-instance архитектуры.**
Rate limiting (`slowapi`, `backend/app/utils/rate_limit.py`) — in-memory, без Redis-бэкенда. Напоминания о записях (`backend/app/scheduler.py`) — in-process `asyncio`-луп без блокировок/leader election. Оба механизма рассчитаны строго на один инстанс бэкенда: горизонтальное масштабирование (2+ реплики backend) без переделки приведёт к сбросу рейтлимитов при рестарте одного из инстансов и к дублирующимся email-напоминаниям.

**#10. `CORS_ORIGINS` не имеет явного прод-оверрайда.**
Дефолт в `backend/app/config.py:127-132` — только localhost. В текущей топологии (nginx/Caddy отдают фронтенд и API same-origin) это не создаёт проблем, но `docker-compose.yml` не переопределяет `CORS_ORIGINS` явно — при любой смене топологии (например, API на отдельном поддомене) браузерный CORS тихо сломается.

**#11. Нет сканирования зависимостей и уязвимостей.**
Ни `pip-audit`, ни `npm audit`, ни Dependabot, ни CodeQL, ни какой-либо image-скан (Trivy/Grype) не настроены в `.github/`.

**#12. ~~Непоследовательная обработка SQL LIKE-паттернов~~ — ложное срабатывание, перепроверено.**
Первоначальный аудит утверждал, что `service_repository.py` не экранирует wildcard-символы поиска через `escape_like()`, в отличие от `user_repository.py`/`master_repository.py`. При прямом чтении файла перед реализацией фикса (Фаза 1) выяснилось, что `service_repository.py:40-42` уже вызывает `escape_like()` точно так же, как остальные два репозитория — исходное утверждение было ошибкой при первичном исследовании кода, не отражало реальное состояние. Действие не требуется, пункт снят.

**#13. Нет линтеров и форматтеров.**
Ни ruff/black/mypy на бэкенде, ни eslint/prettier на фронтенде не сконфигурированы — подтверждено отсутствием конфигов в обеих частях проекта.

**#23. (найдено при реализации Фазы 1) Vite/esbuild dev-server уязвимости, требуют мажорного апгрейда.** `npm audit` показывает 3 находки на связке `vite`/`esbuild` (path traversal в обработке `.map` при `vite optimize`, обход `server.fs.deny`, утечка NTLMv2-хэша через `launch-editor` на Windows) — все специфичны для **dev-сервера** (`npm run dev`), не затрагивают собранную статику, которая реально деплоится. ## 4. Функционал «Сеть парикмахерских» (роль владельца)

### 4.1. Обзор и роли

Новая иерархия ролей: **`owner`** (владелец, вся сеть — новая, самая старшая) → **`admin`** (теперь привязан к одной точке) → **`master`** → **`client`**. Значение enum `admin` не переименовывается — оно просто становится salon-scoped через новую колонку. `master`/`client` как роли не меняются, но `Master` получает обязательную привязку к салону.

**Важно про существующее имя `_ensure_owner_or_admin` в `backend/app/routes/masters.py`** — это helper про *владение ресурсом* («это профиль этого пользователя»), никак не связанный с новой ролью `owner`. Прямая коллизия имён — переименовать в `_ensure_can_manage_master` при реализации (см. §4.8), иначе крайне легко всё перепутать.

Ключевое допущение всей спецификации: **один мастер работает ровно в одной точке**, не заявлено обратное. Если это не так — см. §4.11.

### 4.2. Новая сущность `Salon`

Новая таблица `salons`, новая модель `backend/app/models/salon.py`, миксины `UUIDPrimaryKeyMixin` + `TimestampMixin` (без `SoftDeleteMixin` — точка не удаляется физически, только `is_active=false`, аналогично `Service`/`Master`).

| Колонка | Тип | Nullable | Комментарий |
|---|---|---|---|
| `id` | UUID PK | нет | `uuid.uuid4`, как везде в проекте |
| `name` | `String(150)` | нет | «Сайтама на Тверской» |
| `slug` | `String(150)`, `UNIQUE` | нет | генерируется сервером из `name` (транслитерация + slugify + суффикс при коллизии), не редактируется вручную в v1 |
| `address` | `String(300)` | нет | |
| `phone` | `String(20)` | да | отдельно от `users.phone` |
| `open_time` | `Time` | нет | переезжает из `SiteSettings.business_hours` (см. §4.4) |
| `close_time` | `Time` | нет | |
| `photo_url` | `String(500)` | да | по аналогии с `masters.photo_url`, тот же `POST /api/admin/uploads/image` |
| `is_active` | `Boolean`, default `true` | нет | «закрыть точку» без физического удаления |
| `created_at` | из `TimestampMixin` | нет | `server_default=NOW()` |

Ограничения: `CheckConstraint("close_time > open_time", name="ck_salons_close_after_open")` (тот же паттерн, что уже применён в `BusinessHoursContent.close_after_open` и `ck_schedules_end_after_start`); `UNIQUE` индекс на `slug`.

**Без `DELETE`-эндпоинта** — только `is_active`, т.к. `masters.salon_id`/`appointments.salon_id` ссылаются `ondelete=RESTRICT`, физическое удаление строки `salons` через API недостижимо в принципе.

**Без колонки `timezone`.** Вся система уже жёстко считает «время салона == UTC» минимум в трёх местах (`AppointmentService._validate_within_schedule`, генерация слотов, фронтенд явно форматирует с `timeZone: 'UTC'`). Честная per-salon таймзона — отдельный проект (переписать генерацию слотов и обе валидации рабочих часов), не входит в v1. **Допущение: все точки сети — в одном часовом поясе.**

### 4.3. Где размещается `salon_id`

| Таблица | `salon_id`? | Nullable | `ondelete` | Обоснование |
|---|---|---|---|---|
| `masters` | да | `NOT NULL` | `RESTRICT` | мастер физически работает в одной точке |
| `appointments` | да, денормализовано | `NOT NULL` | `RESTRICT` | тот же паттерн денормализации, что уже применён в `Review` (`master_id`/`service_id` из `Appointment`) — нужно для быстрых per-salon отчётов без JOIN на каждый агрегатный запрос. Проставляется в `AppointmentRepository.create()` из уже загруженного `master.salon_id`. Значение — снимок на момент брони (как `final_price`): перевод мастера в другую точку не переносит старые записи задним числом |
| `users` | да, только для `admin` | `NULLABLE` | `RESTRICT` | см. ниже |
| `schedules` | нет | — | — | наследуется через `master_id → masters.salon_id`, отдельного кросс-салонного запроса по расписаниям нет |
| `services` | нет | — | — | каталог остаётся общим на сеть (см. ниже) |
| `reviews` | нет | — | — | скоуп достижим через существующий JOIN на `Master` (см. ниже) |

**FK, а не M:N-таблица.** Для `admin` — «один admin = одна точка» прямо следует из задачи, junction-таблица была бы избыточной. Для `master` — мультилокационный мастер не заявлен и потребовал бы отдельной модели (per-salon расписание, т.к. `Schedule` сегодня держит ровно одну запись на `(master_id, day_of_week)`) — это отдельная фича, не часть v1.

**`users.salon_id` нужен только для `admin`** (не дублируется на `User` для мастеров — она уже живёт на `Master`, чтобы не было двух источников истины). Инвариант закреплён на уровне БД:
```python
CheckConstraint("role <> 'admin' OR salon_id IS NOT NULL", name="ck_users_admin_requires_salon")
```
плюс проверка в сервисном слое (defense in depth).

**`services.salon_id` — сознательно НЕ добавляется.** Сеть по определению стандартизирует меню/цены; локальная вариативность цены уже полностью покрывается `MasterService.price_override`/`Master.coefficient` (мастер уже привязан к точке). Если реально нужны разные меню по точкам (не только разные цены) — логическое расширение вне v1.

**`reviews.salon_id` — сознательно НЕ добавляется.** Скоупинг модерации отзывов достигается через `JOIN Master ON Master.id = Review.master_id WHERE Master.salon_id = :scope` — этот join уже физически есть в запросе (`joinedload`), нужно только сделать его фильтрующим.

### 4.4. Разделение `SiteSettings`

Решение: **бренд остаётся один на сеть**, «профиль точки» — обычные колонки на `Salon`, не JSONB.

- **Остаётся общим в `SiteSettings.content`**: `header`, `seo`, `hero`, `features`, `services` (текст секции), `masters` (текст секции), `cta`, `theme`. Право `PATCH /api/settings` переходит с `admin` на **`owner`-only** — это управление брендом всей сети.
- **Переезжает из `SiteContent` в `Salon`**: `BusinessHoursContent` (`business_hours.open_time/close_time` → `Salon.open_time/close_time`) и `FooterContent.address`/`FooterContent.hours` (описывают конкретную точку — при 2+ салонах общий адрес в подвале сайта становится некорректным по определению). Поля `address`/`hours` удаляются из `FooterContent` в `backend/app/schemas/site_settings.py`. Поскольку `SiteContent` не имеет `extra="forbid"`, старые ключи в уже сохранённом JSONB просто игнорируются `model_validate` — отдельная миграция для очистки JSONB не обязательна.
- **Публичный сайт без отдельной «страницы выбора локации»**: выбор салона встраивается в существующую воронку `/masters` → `/masters/:id` через новый query-фильтр `salon_id` (тот же паттерн, что уже есть у `specialization`/`service_id`). На `HomePage.vue` добавляется информационная секция «Наши салоны» (новый компонент, тянет `GET /api/salons`, карточки со ссылкой на `/masters?salon_id=...`).

### 4.5. Уникальность полей пользователя

`users.email`/`phone`/`vk_user_id` остаются **глобально уникальными**, без изменений — прямое следствие «один аккаунт на всю сеть»: клиент заходит тем же логином в любую точку. Существующие частичные индексы (`WHERE deleted_at IS NULL`) уже реализуют это корректно.

### 4.6. Защита от двойного бронирования

**Изменений не требуется.** `EXCLUDE USING gist` (`no_double_booking`, `0001_initial.py`) ключуется по `master_id` + пересечению времени. Поскольку один мастер жёстко принадлежит одной точке, физическая невозможность быть в двух местах одновременно уже полностью гарантирована — салон в это уравнение не входит и не должен.

### 4.7. Миграции и бэкфилл

#### ⚠️ Обязательная правка `alembic/env.py` перед миграцией с новым enum-значением

Текущий `backend/alembic/env.py:47-53`:
```python
with connectable.connect() as connection:
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()
```
`context.begin_transaction()` вызывается один раз на весь запуск `alembic upgrade head`, а не на каждую ревизию — все накатываемые за один вызов ревизии выполняются в одной транзакции. PostgreSQL **не позволяет использовать значение enum, добавленное `ALTER TYPE ... ADD VALUE`, в той же транзакции**, где оно было добавлено — даже если `ADD VALUE` и `UPDATE ... SET role='owner'` разнесены по разным файлам миграций, при обычном `alembic upgrade head` они попадут в одну транзакцию, и Postgres кинет `unsafe use of new value "owner" of enum type user_role`.

**Обязательно** добавить `transaction_per_migration=True` в оба вызова `context.configure(...)` (`run_migrations_online` и `run_migrations_offline`) до начала работы над миграцией `0014`. Каждая ревизия коммитится отдельно. Побочный эффект (принятый компромисс): при падении миграции посередине многоревизионного прогона более ранние ревизии этого прогона не откатятся автоматически.

#### Точная последовательность

**`0013_salons`** (`down_revision = "0012_email_verification"`)
- `CREATE TABLE salons (...)` — все колонки §4.2, `CheckConstraint ck_salons_close_after_open`, `UNIQUE INDEX uq_salons_slug`.
- Бэкфилл ровно одной строки «Салон №1»: читаем текущий `site_settings.content` (может отсутствовать на свежей БД — `SiteSettings` создаётся лениво), берём `name` из `content['header']['brand_name']` (иначе `"Салон №1"`), `address` из `content['footer']['address']` (иначе заглушка), часы из `content['business_hours']` (иначе `09:00`/`20:00`). `slug = 'salon-1'` (гарантированно уникален — единственная строка на этом этапе).

4. `op.add_column("masters", sa.Column("salon_id", ...))` (nullable пока)

Новое, salon-scoped значение `admin` начинает выдаваться только вперёд, явным действием owner'а. Безопасно только благодаря тому, что `0014` уже закоммитила `'owner'` отдельной транзакцией.

### 4.8. API-поверхность

**Решение: без отдельного роутера `/api/owner/*`.** Owner получает доступ через существующие `admin`-эндпоинты (он — надмножество admin), плюс один новый ресурс `salons`. Роутеры в проекте организованы по ресурсам, а не по ролям (нет отдельного `/api/master/*` — у мастера просто `/masters/me`) — разделять по URL то, что разделяется по правам, задвоило бы логику.

**Обновлённые зависимости авторизации** (`backend/app/services/auth_service.py`):
```python
get_current_client = require_role(UserRole.client, UserRole.admin, UserRole.owner)
get_current_master = require_role(UserRole.master, UserRole.admin, UserRole.owner)
get_current_admin  = require_role(UserRole.admin, UserRole.owner)   # owner неявно включён
get_current_owner  = require_role(UserRole.owner)
```

**Новый scope-хелпер** (например `backend/app/services/_salon_scope.py`):
```python
def resolve_salon_scope(current_user: User, salon_id_param: UUID | None) -> UUID | None:
    """None = вся сеть (доступно только owner). Для admin — всегда его salon_id;
    чужой salon_id в параметре — явный 403, а не тихая подмена."""
    if current_user.role == UserRole.owner:
        return salon_id_param
    if salon_id_param is not None and salon_id_param != current_user.salon_id:
        raise HTTPException(status_code=403, detail="Доступен только собственный салон")
    return current_user.salon_id
```

#### ⚠️ Обязательная попутная правка — закрывает дыру, которую сама фича делает опасной

Сегодня `AdminUsers.vue`/бэкенд позволяют **любому** admin назначить **любую** роль **любому** пользователю через простой `<select>`, без дополнительных проверок. До этой фичи максимум, чем это грозило — плоское самоназначение admin. После введения иерархии тот же непроверенный код позволил бы salon-admin назначить **себе или кому угодно `owner`** — захват всей сети. Это обязательная часть реализации, не опциональное улучшение:
```python
# AdminService.change_role / create_user
if role in (UserRole.admin, UserRole.owner) and requesting_user.role != UserRole.owner:
    raise HTTPException(403, "Назначать admin/owner может только owner")
```

### 4.11. Явно вне периметра v1

Отмечено как «логическое расширение, подтвердить перед реализацией» — не строить без отдельного запроса:
- **Мастер в нескольких точках одновременно** — потребует per-salon `Schedule` (сейчас одна запись на `(master_id, day_of_week)`), не только M:N на `Master`.
- **Разные каталоги услуг по точкам** (не только разные цены) — потребует `salon_id` на `Service`, взрыв M:N-матрицы `MasterService`.
- **Отдельные часовые пояса по точкам** — переписать генерацию слотов и обе валидации рабочих часов под per-salon timezone вместо глобального UTC-допущения.
- **Церемониальный флоу передачи роли `owner`** — технически `PATCH .../role {role: owner}` от owner уже разрешён на бэкенде, но простой инлайн-дропдаун в `AdminUsers.vue` эту опцию сознательно не предлагает; передача владения сетью заслуживает отдельного флоу с подтверждением.
- **Полноценный SaaS на несколько независимых сетей** (см. также договорённости в начале документа) — потребовал бы дополнительный уровень `Company`/`Tenant` НАД `Salon`, поддомен/слаг-роутинг на уровне запроса, платформенного суперадмина, вероятно — биллинг. Если такая потребность появится позже, текущая модель (`Salon` как плоская сущность внутри одной сети) расширяется добавлением `company_id` на `Salon` и переносом части глобальных сегодня uniqueness-constraints (`users.email` и т.д.) на пару `(company_id, email)` — но это отдельный проект, не инкремент этой спецификации.
